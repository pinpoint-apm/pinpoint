# 코드 리뷰 및 회귀 방지 정책

**이 규칙은 모든 코드 수정에 예외 없이 적용됩니다.**

## 필수 코드 리뷰 프로세스

코드를 수정할 때마다 다음 리뷰를 반드시 수행해야 합니다:

### 1. 수정 전 분석
- 변경 전에 수정할 파일을 완전히 읽기
- 수정되는 코드의 모든 호출자, 임포터, 소비자 파악
- 유지해야 할 기존 동작 이해

### 2. 영향 분석
- 수정된 함수, 컴포넌트, 타입, 훅, 상수의 모든 사용 위치 탐색
- 데이터 흐름 추적: 타입이 변경되면 해당 타입을 생성하거나 소비하는 모든 위치 확인
- 훅을 수정하는 경우, 해당 훅을 사용하는 모든 컴포넌트 확인
- 아톰을 수정하는 경우, 해당 아톰을 읽거나 쓰는 모든 컴포넌트 확인
- 컴포넌트의 props를 수정하는 경우, 해당 컴포넌트를 렌더링하는 모든 부모 컴포넌트 확인
- 엔드포인트 또는 상수를 수정하는 경우, 해당 것을 참조하는 모든 훅과 유틸리티 확인

### 3. 수정 후 검증
- 수정된 코드와 영향 받은 모든 파일 다시 읽기
- 기존 함수 시그니처, 반환 타입, props 인터페이스가 호환성을 유지하는지 확인
- 기존 임포트 경로가 손상되지 않았는지 검증
- 파라미터의 기본값 및 선택적/필수 상태가 유지되는지 확인
- 다른 곳에서 사용되는 i18n 키가 모든 참조를 업데이트하지 않은 채로 제거되거나 이름이 변경되지 않았는지 확인

### 4. 회귀 체크리스트
- [ ] 수정된 함수의 모든 기존 호출자가 올바른 인자를 받음
- [ ] 수정된 타입의 모든 기존 소비자가 타입 계약을 만족함
- [ ] 수정된 컴포넌트의 모든 기존 렌더러가 유효한 props를 전달함
- [ ] 익스포트가 실수로 제거되거나 이름이 변경되지 않음
- [ ] 쿼리 키가 캐시 무효화를 손상시키는 방식으로 변경되지 않음
- [ ] 라우트 로더가 여전히 올바르게 검증하고 리다이렉트함
- [ ] InitialFetchOutlet이 URL 파라미터를 아톰에 올바르게 동기화하고 설정 오류를 처리함

### 5. 테스트 인식
- 테스트가 있는 코드를 수정하는 경우 관련 테스트 실행: `yarn workspace @pinpoint-fe/ui test`
- 테스트가 없는 코드를 수정하는 경우 위험도를 고려하고 핵심 경로에 테스트 추가를 제안
- 다음 코드 변경 시 절대 테스트 검증을 건너뛰지 않음: 아톰, 유틸리티 함수, API 훅, 라우트 로더

## 리뷰 범위
- **단일 파일 변경**: 해당 파일 + 모든 직접 임포터 리뷰
- **파일 간 변경**: 변경 세트의 모든 파일 + 해당 임포터 리뷰
- **타입/인터페이스 변경**: 해당 타입을 참조하는 모든 파일 리뷰 (Grep 사용)
- **상수/엔드포인트 변경**: 해당 상수를 사용하는 모든 훅과 컴포넌트 리뷰
